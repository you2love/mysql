SQL原理

1. 索引概述

索引是数据库中用于加速数据检索的特殊数据结构。在 MySQL 中，索引本质上是帮助快速定位数据的数据结构，类似于书籍的目录。理解索引的实现原理对于设计高性能数据库至关重要。

2. B+ Tree 索引原理

2.1 B+ Tree 结构

B+ Tree 是 MySQL 中 InnoDB 存储引擎默认使用的索引结构。它是一种自平衡的多路搜索树，所有数据都存储在叶子节点，非叶子节点只存储索引键。

-- 1. 非叶子节点只存储索引键（冗余存储），不存储实际数据
-- 2. 叶子节点存储所有数据（包含主键和所有列）
-- 3. 叶子节点之间通过双向链表连接
-- 4. 所有叶子节点深度相同，查询时间复杂度稳定 O(log n)

-- 假设 B+ Tree 阶数 m=4：
--        [50]              非叶子节点（索引键）
--      /   |   \
--   [10] [30] [50,70,90]  叶子节点（数据+链表）
--    |    |    |
--   ...  ...  ...

2.2 B+ Tree 查询过程

-- 查询 id=50 的记录过程：
-- 1. 从根节点开始，50 >= 50，进入中间子树
-- 2. 比较 30 < 50 < 70，进入左侧叶子节点
-- 3. 在叶子节点中顺序查找，找到目标记录

-- B+ Tree 查询效率：
-- - 高度通常为 2-3 层（百万级数据）
-- - 每次 IO 可读取一个节点（约 16KB）
-- - 磁盘 IO 次数 = 树高度 ≈ 3 次

-- 对比全表扫描：需要读取所有数据页

2.3 B+ Tree vs B Tree

-- B+ Tree vs B Tree 对比：
特性           | B+ Tree              | B Tree
---------------|--------------------|----------------
数据存储       | 只在叶子节点        | 所有节点都存储数据
查询稳定性     | O(log n)，所有查询 | O(log n)，不同位置
范围查询        | 叶子节点链表，高效     | 需要遍历树
磁盘 IO        | 更少（只读叶子节点）| 更多
空间利用率     | 更高（节点满）      | 相对较低

-- 为什么 MySQL 选择 B+ Tree？
-- 1. 范围查询更高效，适合 ORDER BY
-- 2. 查询效率更稳定（都要到叶子节点）
-- 3. 更适合磁盘存储，IO 次数少

3. InnoDB 数据页存储原理

3.1 数据页概述

InnoDB 是 MySQL 默认的存储引擎，其数据存储的基本单位是数据页（Page）。每个数据页默认大小为 16KB，是 InnoDB 与磁盘交互的最小单位。理解数据页的结构对于深入理解 MySQL 的索引机制和性能优化至关重要。

-- InnoDB 数据页基本参数：
-- - 页大小: 16KB (可通过 innodb_page_size 配置)
-- - 页类型: 数据页、索引页、Undo页、系统页等
-- - 磁盘IO: 每次读写以页为单位
-- - 缓存: 页数据缓存在 Buffer Pool 中

-- 查看数据页大小：
SHOW VARIABLES LIKE 'innodb_page_size';

-- 典型输出：16384 (16KB)

3.2 数据页结构

InnoDB 数据页由以下部分组成，每个部分都有固定的大小和作用：

3.3 各组成部分详解

-- 1. File Header (文件头) - 38字节
--    包含：页ID、上一页/下一页指针、页类型、checksum等
--    作用：管理页之间的关联关系

-- 2. Page Header (页头) - 56字节
--    包含：已记录数、空闲空间起始位置、页目录数量等
--    作用：记录页的元数据信息

-- 3. Infimum + Supremum (虚拟记录) - 各13字节
--    Infimum：比任何用户记录都小的伪记录
--    Supremum：比任何用户记录都大的伪记录
--    作用：作为页内记录的边界标记

-- 4. User Records (用户记录区)
--    实际存放的行数据，采用行格式存储
--    记录通过 Free Space 區域动态分配

-- 5. Free Space (空闲空间)
--    未使用的空间，用于插入新记录
--    随着记录增加而减少

-- 6. Page Directory (页目录) - 目录项数组
--    存放记录指针，类似于书的目录
--    作用：二分查找，快速定位记录
--    每个目录项约 2 字节

-- 7. File Trailer (文件尾) - 8字节
--    存储页的checksum和LSN
--    作用：检测页是否完整损坏

3.4 数据页如何存储记录

InnoDB 使用行格式（Row Format）存储记录，主要有 COMPACT、DYNAMIC、COMPRESSED 等格式。下面以 COMPACT 格式为例说明：

-- COMPACT 行格式存储结构：
-- +------------------------------+
-- | 变长字段长度列表             | (1-2字节/字段)
-- | NULL值列表                   | (1字节/字段，可NULL)
-- | 记录头信息                   | (5字节)
-- | 字段1值                      |
-- | 字段2值                      | (实际数据)
-- | ...                          |
-- +------------------------------+

-- 示例：存储用户表记录
CREATE TABLE users (
    id INT PRIMARY KEY,        -- 4字节
    name VARCHAR(50),        -- 可变长度
    age INT,                   -- 4字节
    email VARCHAR(100)      -- 可变长度
);

-- 一条实际存储的记录可能像这样：
-- [变长长度: 8,4,7] [NULL: 无] [头信息] [id=1] [name=zhangsan] [age=30] [email=test@t.com]

3.5 索引页与数据页的关系

InnoDB 中有两种重要的页类型：索引页（Index Page）和数据页（Data Page）。在 B+ Tree 中：

-- 索引页 vs 数据页：

-- 索引页（Non-leaf Page）：
-- - 存储索引键和子页指针
-- - 不存储实际数据行
-- - 格式：[主键, 页指针], [主键, 页指针], ...
-- - 示例：存储 [20 -> 页A], [40 -> 页B], [60 -> 页C]

-- 数据页（Leaf Page）：
-- - 存储完整的行数据
-- - 主键索引的叶子节点：存储完整行数据
-- - 辅助索引的叶子节点：存储主键值（需要回表）
-- - 叶子节点之间通过双向链表连接

-- 示例：B+ Tree 索引结构
--            [50]               <- 索引页（根）
--        /    |    \
--    [10,30] [50,70] [90]     <- 索引页（中间层）
--    / | \    / | \    / | \
--  数据 数据 数据 数据 数据    <- 数据页（叶子节点）

3.6 数据页的查找过程

当执行 SQL 查询时，InnoDB 如何从数据页中找到目标记录：

-- 数据页内查找记录的过程：

-- 1. 通过 B+ Tree 定位到目标数据页
--    磁盘IO: 读取目标页到 Buffer Pool

-- 2. 在数据页内查找记录：
--    a) 使用 Page Directory 进行二分查找
--    b) 定位到记录所在的 Slot
--    c) 顺序查找定位精确记录

-- 3. 返回记录（或通过主键回表获取完整数据）

-- 示例：查询 WHERE id = 25
SELECT * FROM users WHERE id = 25;

-- 执行过程：
-- 1. 在索引树中找到 id=25 所属的叶子页
-- 2. 将页加载到内存（Buffer Pool）
-- 3. 在页内通过 Page Directory 快速定位
-- 4. 读取完整记录返回

-- 性能优化提示：
-- - 尽量使用覆盖索引，避免回表
-- - 减少页的磁盘IO，提高缓存命中率
-- - 合理设置 Buffer Pool 大小

3.7 数据页与 Buffer Pool

InnoDB 使用 Buffer Pool（缓冲池）来缓存数据页，减少磁盘 IO，提高查询性能。

-- Buffer Pool 原理：

-- 1. 数据页缓存
--    - 读取数据时，先从 Buffer Pool 查找
--    - 未命中时，从磁盘读取并缓存
--    - 减少磁盘 IO 操作

-- 2. 脏页管理
--    - 修改的数据页标记为"脏页"
--    - 后台线程定期刷新到磁盘
--    - 脏页比例影响性能

-- 3. LRU 淘汰策略
--    - 使用改进的 LRU 算法管理缓存
--    - 新读取的页插入到链表头部
--    - 冷数据被淘汰出 Buffer Pool

-- 查看 Buffer Pool 状态：
SHOW ENGINE INNODB STATUS;

-- 查看 Buffer Pool 大小配置：
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
-- 建议配置为可用内存的 50%-70%

-- Buffer Pool 命中率计算：
-- Hit Rate = (1 - Key_reads / Key_requests) * 100%
-- 命中率应该 > 99%

4. 哈希索引原理

3.1 哈希表结构

哈希索引使用哈希表实现，通过哈希函数将索引键转换为哈希值，然后用哈希值定位数据。哈希索引的查询时间复杂度为 O(1)，但只支持精确匹配查询。

-- 哈希索引结构：
-- 哈希函数：hash(key) -> hash_value
-- 哈希表：bucket -> 指向数据行

-- 示例：
key = 'zhangsan'
hash('zhangsan') = 0x7f3a (哈希值)
bucket[0x7f3a] -> [row1, row2] (哈希冲突链表)

-- 创建哈希索引（Memory/Heal 存储引擎）
CREATE TABLE t1 (
    id INT,
    name VARCHAR(100),
    INDEX idx_name (name) USING HASH
) ENGINE=MEMORY;

3.2 哈希索引的限制

-- 哈希索引的限制：
-- 1. 只支持等值查询，不支持范围查询
SELECT * FROM users WHERE age > 25;  -- ❌ 无法使用哈希索引
SELECT * FROM users WHERE name = 'zhangsan';  -- ✅ 可以使用

-- 2. 不支持排序和前缀匹配
SELECT * FROM users ORDER BY name;  -- ❌ 无法使用
SELECT * FROM users WHERE name LIKE 'zhang%';  -- ❌ 无法使用

-- 3. 哈希冲突导致性能下降
-- 4. 无法避免回表查询

-- InnoDB 自适应哈希索引 (AHI)
-- InnoDB 会自动为热点数据创建哈希索引
SHOW ENGINE INNODB STATUS;  -- 查看自适应哈希索引状态

4. 全文索引原理

4.1 倒排索引结构

全文索引采用倒排索引（Inverted Index）结构，与正向索引相反，倒排索引记录每个词条出现在哪些文档中。这使得全文搜索非常高效。

-- 倒排索引结构：
-- 正向索引：文档 -> [词条1, 词条2, ...]
-- 倒排索引：词条 -> [文档1, 文档2, ...]

-- 示例文档：
Doc1: "MySQL is a database"
Doc2: "MySQL is open source"

-- 倒排索引：
"MySQL"   -> [Doc1, Doc2]
"is"      -> [Doc1, Doc2]
"database" -> [Doc1]
"open"     -> [Doc2]
"source"   -> [Doc2]

-- 创建全文索引
CREATE TABLE articles (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,
    FULLTEXT INDEX ft_idx (title, content) WITH PARSER ngram
) ENGINE=InnoDB;

4.2 全文搜索机制

-- 全文搜索模式：
-- 1. NATURAL LANGUAGE MODE（自然语言模式）
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('database' IN NATURAL LANGUAGE MODE);

-- 2. BOOLEAN MODE（布尔模式）
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('+MySQL -database' IN BOOLEAN MODE);

-- 3. QUERY EXPANSION（查询扩展）
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('MySQL' WITH QUERY EXPANSION);

-- 4. ngram 分词器（中文）
CREATE FULLTEXT INDEX ft_zh ON articles(content) WITH PARSER ngram;

5. 空间索引原理

5.1 R-Tree 结构

空间索引使用 R-Tree（区域树）结构，它是一种专门用于处理多维数据的树结构。R-Tree 将空间划分为层次嵌套的矩形区域，能够高效处理空间查询。

-- R-Tree 结构原理：
-- 1. 空间被递归划分为嵌套的最小边界矩形 (MBR)
-- 2. 每个节点存储子节点的 MBR
-- 3. 叶子节点存储实际的空间对象

-- R-Tree 示例：
--        [MBR1]           根节点
--      /   |   \
--  [MBR2] [MBR3] [MBR4]  中间节点
--   / \    |     / \
-- P1  P2  P3   P4  P5    叶子节点（空间对象）

-- 创建空间索引
CREATE TABLE locations (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    location GEOMETRY NOT NULL,
    SPATIAL INDEX idx_location (location)
) ENGINE=InnoDB;

5.2 空间查询类型

-- 空间查询示例：
-- 1. 包含查询：查找多边形内的点
SELECT * FROM locations
WHERE ST_Contains(
    ST_GeomFromText('POLYGON((0 0, 10 0, 10 10, 0 10, 0 0))'),
    location
);

-- 2. 距离查询：查找距离某点一定范围内的对象
SELECT *, ST_Distance_Sphere(location, ST_GeomFromText('POINT(116.4 39.9)'))
FROM locations
WHERE ST_Distance_Sphere(location, ST_GeomFromText('POINT(116.4 39.9)')) < 50000;

-- 3. 相交查询：查找相交的空间对象
SELECT * FROM locations
WHERE ST_Intersects(location, ST_GeomFromText('LINESTRING(0 0, 10 10)'));

6.1 聚簇索引（Clustered Index）

聚簇索引决定了数据的物理存储顺序。每个表只能有一个聚簇索引，因为数据只能按一种方式物理排序。通常主键就是聚簇索引。

-- InnoDB 聚簇索引特点：
-- 1. 数据行直接存储在叶子节点中
-- 2. 主键索引就是聚簇索引
-- 3. 如果没有主键，使用第一个唯一非空索引
-- 4. 如果没有唯一索引，InnoDB 会生成隐藏的 ROW_ID

-- 聚簇索引结构：
-- 叶子节点存储完整的数据行
-- +--------+-------------+-----+-----+
-- | id (PK)| name        | age | ... |
-- +--------+-------------+-----+-----+
-- | 1      | zhangsan    | 25  | ... |
-- | 2      | lisi        | 30  | ... |
-- | 3      | wangwu      | 28  | ... |
-- +--------+-------------+-----+-----+

CREATE TABLE users (
    id INT PRIMARY KEY,  -- 聚簇索引
    name VARCHAR(100),
    age INT,
    INDEX idx_age (age)   -- 非聚簇索引
);

6.2 非聚簇索引（Secondary Index）

-- 非聚簇索引（辅助索引）特点：
-- 1. 叶子节点存储索引键 + 主键值
-- 2. 一个表可以有多个非聚簇索引
-- 3. 查询时需要先通过非聚簇索引找到主键
-- 4. 再通过主键到聚簇索引中获取完整数据（回表）

-- 非聚簇索引 idx_age 结构：
-- +--------+--------+
-- | age    | id(PK) |
-- +--------+--------+
-- | 25     | 1      |
-- | 28     | 3      |
-- | 30     | 2      |
-- +--------+--------+

-- 查询 SELECT * FROM users WHERE age = 25：
-- 1. 在 idx_age 中找到 age=25 的记录，获取 id=1
-- 2. 在聚簇索引中找到 id=1 的完整数据
-- 3. 返回完整数据行

6.3 覆盖索引避免回表

-- 覆盖索引：索引包含查询所需的所有列
-- 不需要回表，直接从索引中返回数据

-- 创建覆盖索引
CREATE INDEX idx_age_name ON users(age, name);

-- 使用覆盖索引（不需要回表）
SELECT age, name FROM users WHERE age > 25;
-- EXPLAIN: Extra = Using index

-- 需要回表的查询
SELECT * FROM users WHERE age > 25;
-- EXPLAIN: Extra = Using index condition

-- 验证是否使用覆盖索引
EXPLAIN SELECT age, name FROM users WHERE age = 25;
-- type=ref, key=idx_age_name, Extra=Using index

6.4 回表原理详解

回表是指查询过程中，先通过非聚簇索引找到主键值，再根据主键值到聚簇索引中查找完整数据行的过程。这是 InnoDB 存储引擎的重要特性。

-- 回表的完整过程：
-- 1. 在非聚簇索引树中查找
-- 2. 获取满足条件的主键值
-- 3. 使用主键值在聚簇索引中查找完整数据
-- 4. 返回完整的用户数据

-- 示例表结构：
CREATE TABLE orders (
    id INT PRIMARY KEY,           -- 聚簇索引
    order_no VARCHAR(20),       -- 唯一索引
    customer_id INT,
    amount DECIMAL(10,2),
    status VARCHAR(10),
    created_at DATETIME,
    INDEX idx_customer (customer_id) -- 非聚簇索引
);

-- 执行 SELECT * FROM orders WHERE customer_id = 100
-- 1. 在 idx_customer 中找到 customer_id=100 的记录
-- 2. 获取主键 id（如 id=500）
-- 3. 在聚簇索引中查找 id=500 的完整记录
-- 4. 返回所有列数据

EXPLAIN SELECT * FROM orders WHERE customer_id = 100;
-- type: ref, key: idx_customer
-- Extra: Using index condition (需要回表)

6.5 回表性能影响

-- 回表的性能影响：
-- 1. 增加磁盘 IO：每次回表可能产生一次磁盘读取
-- 2. 随机 IO：聚簇索引中查找是随机 IO
-- 3. 回表次数 = 索引命中的记录数

-- 回表次数过多的例子
SELECT * FROM orders WHERE customer_id > 100;
-- 假设 customer_id > 100 有 10000 条记录
-- 需要回表 10000 次，性能很差

EXPLAIN SELECT * FROM orders WHERE customer_id > 100;
-- type: range, key: idx_customer
-- rows: 10000 (预计回表次数)

-- 优化方案：使用覆盖索引
CREATE INDEX idx_cover ON orders(customer_id, order_no, amount, status);

-- 如果查询的列都在索引中，不需要回表
SELECT customer_id, order_no, amount, status 
FROM orders WHERE customer_id > 100;

EXPLAIN SELECT customer_id, order_no, amount, status 
FROM orders WHERE customer_id > 100;
-- Extra: Using index (覆盖索引，无需回表)

6.6 减少回表的方法

-- 方法1：使用覆盖索引
-- 将需要查询的列都包含在索引中

-- 方法2：优化查询条件
-- 避免返回大量数据
SELECT id, order_no FROM orders WHERE customer_id = 100
LIMIT 100;  -- 限制返回数量

-- 方法3：使用主键查询
SELECT * FROM orders WHERE id = 500;
-- 直接在聚簇索引中查找，不需要回表

-- 方法4：分页优化
-- 避免 OFFSET 过大导致的深度分页问题

-- 低效（回表次数多）：
SELECT * FROM orders ORDER BY id LIMIT 10000, 10;

-- 高效（基于主键范围）：
SELECT * FROM orders WHERE id > 10000 LIMIT 10;
-- 上一页最后一条记录的 id 作为起点

-- 方法5：使用延迟关联
SELECT o.* 
FROM orders o 
INNER JOIN (
    SELECT id FROM orders WHERE customer_id = 100
    LIMIT 100
) t ON o.id = t.id;

7. 索引存储结构

7.1 InnoDB 页面结构

-- InnoDB 数据页结构（默认 16KB）：
-- +------------------+
-- | File Header      | 38 bytes - 文件头
-- +------------------+
-- | Page Header      | 56 bytes - 页头
-- +------------------+
-- | Infimum + Supremum| 26 bytes - 虚拟行
-- +------------------+
-- | User Records     | 变长 - 用户数据
-- +------------------+
-- | Free Space       | 变长 - 空闲空间
-- +------------------+
-- | Page Directory   | 变长 - 页目录
-- +------------------+
-- | File Trailer     | 8 bytes - 文件尾
-- +------------------+

-- 页内记录存储：
-- 1. 记录头信息（变长字段长度、NULL标志等）
-- 2. 记录实际数据
-- 3. 记录通过 Page Directory 实现二分查找

7.2 索引页结构

-- B+ Tree 索引页结构：
-- +------------------+
-- | File Header      | 38 bytes
-- +------------------+
-- | Page Header      | 56 bytes
-- +------------------+
-- | User Records     | 索引记录
-- |  (非叶子节点)    | (最小键, 指向子页指针)
-- |  (叶子节点)      | (键值, 主键/行指针)
-- +------------------+
-- | Free Space       | 空闲空间
-- +------------------+
-- | Page Directory   | 槽位目录（记录分组）
-- +------------------+
-- | File Trailer     | 8 bytes
-- +------------------+

-- 非叶子节点记录格式：
-- | 记录头 | 索引键值 | 页面指针 |

-- 叶子节点记录格式（聚簇索引）：
-- | 记录头 | 主键值 | 其他列数据 |

-- 叶子节点记录格式（非聚簇索引）：
-- | 记录头 | 索引键值 | 主键值 |

8. 索引优化原理

8.1 最左前缀原则

-- 复合索引 (a, b, c) 的使用规则：
WHERE a = 1                 -- ✅ 使用索引 a
WHERE a = 1 AND b = 2        -- ✅ 使用索引 a, b
WHERE a = 1 AND b = 2 AND c = 3  -- ✅ 使用索引 a, b, c
WHERE b = 2                 -- ❌ 不使用索引
WHERE b = 2 AND c = 3    -- ❌ 不使用索引
WHERE a = 1 AND c = 3    -- ⚠️ 只使用索引 a

-- 原理：复合索引按从左到右的顺序构建 B+ Tree
-- 必须先使用前面的列，才能利用后续列的索引

8.2 索引下推（Index Condition Pushdown）

-- 索引下推原理：
-- 在 MySQL 5.6+ 之前：
-- 1. 通过索引找到满足 a=1 的主键
-- 2. 回表获取完整数据
-- 3. 在 server 层过滤 b>100 的条件

-- MySQL 5.6+ 索引下推：
-- 1. 通过索引找到满足 a=1 的主键
-- 2. 在存储引擎层直接过滤 b>100 的条件
-- 3. 只对满足所有条件的记录回表

-- 示例索引：(a, b)
SELECT * FROM t WHERE a = 1 AND b > 100;

-- 验证索引下推
EXPLAIN SELECT * FROM t WHERE a = 1 AND b > 100;
-- Extra: Using index condition (使用了索引下推)

8.3 索引合并（Index Merge）

-- 索引合并原理：
-- MySQL 可以同时使用多个索引
-- 然后合并结果集

-- 索引合并类型：
-- 1. intersection（交集）- 多个索引结果取交集
-- 2. union（并集）- 多个索引结果取并集
-- 3. sort_union（排序并集）- 先排序再取并集

-- 示例：
SELECT * FROM users WHERE age = 25 AND name = 'zhangsan';

EXPLAIN SELECT * FROM users WHERE age = 25 AND name = 'zhangsan';
-- type: index_merge
-- key: idx_age, idx_name
-- Extra: Using intersect(idx_age,idx_name)

9. 索引实践分析

9.1 使用 EXPLAIN 分析索引

-- EXPLAIN 关键字段：
-- type: 连接类型（从好到差）
--   system > const > eq_ref > ref > range > index > ALL
--   const: 主键/唯一索引等值查询
--   ref: 非唯一索引等值查询
--   range: 索引范围查询
--   index: 全索引扫描
--   ALL: 全表扫描（需要优化）

-- key: 实际使用的索引
-- rows: 预计扫描的行数
-- Extra: 额外信息
--   Using index: 使用覆盖索引
--   Using where: 使用 WHERE 条件过滤
--   Using index condition: 使用索引下推
--   Using filesort: 需要额外排序
--   Using temporary: 使用临时表

EXPLAIN SELECT * FROM users WHERE age = 25;

-- 分析结果：
-- type: ref (使用了非唯一索引)
-- key: idx_age
-- rows: 10 (预计扫描 10 行)

9.2 索引设计最佳实践

-- 索引设计原则：
-- 1. 区分度高的列放在前面
CREATE INDEX idx_status_user ON orders(user_id, status);
-- ✅ user_id 区分度高，放前面

-- 2. 考虑查询的 WHERE 条件顺序
CREATE INDEX idx_age_name ON users(age, name);
-- 适合: WHERE age = ? 或 WHERE age = ? AND name = ?

-- 3. 控制索引数量
-- - 索引占用空间
-- - 影响 INSERT/UPDATE/DELETE 性能
-- - 建议单个表索引数 < 5

-- 4. 使用覆盖索引减少回表
CREATE INDEX idx_user_cover ON orders(user_id, status, created_at);
-- 覆盖: SELECT user_id, status FROM orders WHERE user_id = ?

-- 5. 避免在区分度低的列上建索引
-- 如: 性别、状态（0/1）等
