1. B+ Tree 索引原理

1.1 B+ Tree 结构特点

B+ Tree 是 MySQL 中 InnoDB 存储引擎默认使用的索引结构。它是一种自平衡的多路搜索树，所有数据都存储在叶子节点，非叶子节点只存储索引键。

1.3 B+ Tree 优缺点分析

-- B+ Tree 优点：

-- 1. 查询时间复杂度稳定 O(log n)
--    无论查询什么数据，路径长度相同

-- 2. 范围查询高效
--    叶子节点双向链表，顺序访问快
--    SELECT * FROM users WHERE age BETWEEN 20 AND 30

-- 3. 磁盘 IO 次数少
--    非叶子节点不存储数据，每页可存更多键
--    16KB / 12字节 ≈ 1365 个键值

-- 4. 适合区间统计
--    ORDER BY、GROUP BY 效率高

-- 5. 缓存命中率高
--    非叶子节点可缓存在内存中


-- B+ Tree 缺点：

-- 1. 写入性能较差
--    插入/删除需要维护树的平衡
--    可能触发页分裂/合并

-- 2. 空间占用较大
--    叶子节点存储重复的主键值
--    主键越长，冗余越大

-- 3. 不适合唯一性要求高的场景
--    需要额外维护唯一性

-- 4. 索引列更新成本高
--    索引列值变化需要更新索引

1.4 数据量与索引设计建议

-- 不同数据量的索引设计：

    -- 小数据量 (< 10万)
    -- - 简单索引即可，无需过度优化
    -- - 优先考虑查询便利性
    CREATE INDEX idx_name ON users(name);

    -- 中等数据量 (10万 - 1000万)
    -- - 考虑复合索引覆盖查询
    -- - 关注索引选择性
    CREATE INDEX idx_user_status ON orders(user_id, status);

    -- 大数据量 (> 1000万)
    -- - 考虑分区表
    -- - 使用覆盖索引避免回表
    -- - 定期分析索引效率
    CREATE INDEX idx_cover ON logs(device_id, event_type, created_at);

    -- 超大数据量 (> 1亿)
    -- - 考虑分库分表
    -- - 使用近似查询代替精确查询
    -- - 考虑 ClickHouse/ES 等分析型数据库
